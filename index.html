この資料の使い方

原典のチャプロ社カタログPDFは「ドキュメント作成／アイデア創出／業務効率化…」など 機能別 に60項目が並んでいますが、それでは「自分の事務所のどの場面で使えるか」が即座につながりません。
この資料では、職員3名規模の建築設計事務所の 業務フロー（受注前〜基本設計〜実施設計〜監理〜引渡〜経営）に沿って10カテゴリに再分類し、各カテゴリで「何ができるか」「何を使うか」「どこまでなら自前でできて、どこから外注か」を一望できる構成にしました。

読み方の3つの軸

軸A：課題カテゴリ 業務フロー順10軸（C1〜C10）。困っている業務に直接ジャンプして読む。
軸B：ツール群 A群＝汎用AI：ChatGPT（個人/Team）／Google Workspace Business Standard（Gemini・NotebookLM・Google Apps Script）
B群＝Claude Codeデスクトップ：ローカルファイル直接処理／独自スキル開発／長文・複数ファイル横断処理
軸C：実装難易度 Lv1初心者〜中級者：プロンプト1個・既存UIの操作で完結
Lv2中級〜上級者：定型ワークフロー化・チーム共有
Lv3上級または外注：カスタムチャットボット・RAG・CAD自動化・BIM連携

10カテゴリ詳細

各カテゴリで「誰の悩み／具体ユースケース／推奨ツール（汎用AI／Claude Code）／難易度／日本国内の事例／注意点」を整理。クリックで展開。

誰の悩み？代表が「この敷地・施主に何を提案するか」を言語化するフェーズ。プレゼン骨子作りに時間がかかる、コンセプトのストーリーが弱い、毎回ゼロから書く。
具体ユースケース
  • 敷地条件＋施主要望からプレゼン資料の構成案を5パターン生成
  • 土地の歴史・風土からコンセプトストーリーを言語化（10案 → 3案に絞る）
  • 「住む人」のペルソナ詳細（年齢・職業・家族構成・1日の動線）を物語化
  • 空間の暮らしイメージをコピーライティング化（「リビングは家族の本棚」など）
  • 建物名・マンション名・架空の街区名のネーミング案
  • 施主のライフスタイル別の設計訴求軸（共働き／二世帯／DINKS等）
推奨ツール（A群） ChatGPT（Plus/Team）：単発のコンセプト生成・コピー案出し。GPT-5の長文出力で骨子→詳細展開を一気通貫。
NotebookLM：過去の自分の提案書PDFを20本投入し、「うちの事務所らしいコンセプトの書き方」を学習させて文体を継承させる使い方が有効。
推奨ツール（B群） Claude Codeデスクトップ：過去案件フォルダのプレゼン.pptx・docx・図面を全部読ませて「この事務所の典型コンセプトパターン」を抽出し、新規案件向けのドラフトを一気に書かせる。長文＋複数ファイル横断処理がGPTより得意。
難易度 Lv1 ChatGPTにプロンプト直打ち（5分）。代表が一人で運用可能。
Lv2 「敷地情報→コンセプト案5本」の自前テンプレを社内共有（チームメンバー誰でも同品質）。
Lv3 過去案件のRAG（NotebookLM 50本以上 or 独自RAG）でブランド文体を継承する仕組みを構築。
日本国内の事例

注意点施主の個人情報（氏名・住所・家族構成）をChatGPT個人版に入れない。Plus以上でも学習オプトアウト設定を確認。Claude Codeはローカル処理なのでこの問題に強い。
誰の悩み？設計実務中の「品質チェック」「仕様書の表記揺れ」「複数案の比較」が属人化。代表しか正解を知らない。
具体ユースケース
  • 設計図書の標準化ルール（事務所固有）を文書化
  • 仕様書の用語ゆれ統一（「コンクリート」「RC」「鉄筋コンクリート」の併用整理）
  • 外壁・内装材の色組み合わせ案を画像生成AIで複数パターン
  • 複数の設計案を法規制・周辺環境・コスト軸で客観比較
  • 図面と仕様書の整合性チェック（部屋名・面積・仕上の不一致発見）
  • 過去のトラブル事例・工期遅延データから現場リスク予測
推奨ツール（A群） ChatGPT：仕様書の用語統一・整合性チェック（PDFアップロード→「表記揺れを抽出して」）。
Gemini：Google Drive上の仕様書・図面リストを横断検索・要約。

Lv2 Stable Diffusion + ControlNetでスケッチ→パース、Inpaintでボリューム検討（独学2〜4週間）。

推奨ツール（A群） NotebookLM（最有力）：建築基準法本文PDF・自治体条例PDFを投入し、「敷地は〇〇区××町、用途地域△△の場合の建蔽率は？」と日本語質問。2025年に改正建築基準法を読ませて社内シェアしている実例あり（X上で公開）。

難易度 Lv1 NotebookLMに法令PDFを投入し質問（30分でセットアップ）。
Lv2 自治体条例の常時更新運用ルール化、社内共有Notebook。
Lv3 自前の建築基準法RAGチャットボット構築（GPT4All + LangChain でローカル運用＝API費用ゼロ、note記事に手順公開あり）。

注意点AIの法規解釈は下調べ用。確認申請の最終判断は必ず一級建築士の責任。複雑な法令解釈・行政協議事項は専門家確認が必要。
誰の悩み？議事録に時間がかかる、施主向けの説明が専門用語だらけになる、ヒアリングで聞き漏らす、似たような質問が繰り返し来る。

  • 施主・協力会社・行政向けに文体を切り替えた書類（同じ内容を3バージョン）
  • ヒアリングの質問リスト精緻化（家族構成→潜在ニーズ予測）

FLOW 1 / 議事録AI 4ステップ

★ STEP 3 がAIの主戦場。手書きメモを取らずに録音に集中できるため、打合せ自体の質も上がる。

誰の悩み？設計受託契約・工事監理契約のドラフトに時間がかかる、不利な条項を見落とす、海外の英文契約が読みづらい、資材高騰のリスク分析が手薄。
具体ユースケース
  • 設計受託契約・工事監理契約・特約事項のドラフト生成（弁護士監修テンプレを下敷きに）
  • 受領した契約書から「不利な条項」を抽出（瑕疵担保責任の重さ・著作権帰属・解除条項）
  • 海外プロジェクトの英文契約と日本語契約の対応条項比較
  • 過去のトラブル事例から類似リスク予測
  • 資材高騰・サプライチェーン停滞のシナリオ分析と代替策
推奨ツール（A群） ChatGPT：契約書PDF読み込み→「クライアント有利／受託者有利の条項を分類」「不利な条項に修正案」。
NotebookLM：過去の自社契約書を全部投入し「うちの定型修正パターン」をベース化。
推奨ツール（B群） Claude Codeデスクトップ：契約書フォルダを丸ごと読ませて版間比較（v1→v2→v3でどこが変わったか）。AS資産の /contract-check スキルのようにExcel新旧対比表を自動生成可能。
難易度 Lv1 ChatGPTに契約書PDF＋立場（受託者）→ 不利条項チェック（10分）。
Lv2 自社の契約書テンプレを学習させた専用GPTs構築。
Lv3 顧問弁護士＋AI併用の契約レビューフロー設計（外注／弁護士相談）。
日本国内の事例

  • AS自前スキル：/contract-check（立場別レビュー）・/biz-contract（業務委託契約自動生成）・/doc-compare（新旧対比）を建築事務所向けにカスタマイズ可能。
注意点AIの契約レビューは論点抽出までが限界。最終判断は弁護士に。守秘義務のある契約書はChatGPT個人版・無料版に入れない。
誰の悩み？完成見学会の集客が伸びない、SNS発信が続かない、ブログ記事が書けない、建築写真のキャプションがワンパターン、設計思想を言語化できない。
具体ユースケース
  • 設計思想・施工事例の裏話をブログ記事化（音声録音→AI整文）
  • 建築写真のInstagram/Pinterestキャプション（情緒的・物語性）
  • 完成見学会の集客戦略（DMフローチャート・告知文）
  • 来場予約・設計相談のLP説明文・キャッチコピー
  • 「ZEH住宅」「リノベ」等のトレンドと自社の強みを掛け合わせたポジショニング案
  • メルマガ・LINE公式アカウントの月次配信テンプレ
推奨ツール（A群） ChatGPT：ブログ・SNS・メルマガの文体テンプレを作って量産。
Gemini + Google Workspace：Drive上の過去ブログを学習させて文体継承。
画像生成AI（Midjourney/Stable Diffusion）：SNS用ヘッダー・アイキャッチ。

Lv2 月次SNSカレンダー＋投稿テンプレ化（誰でも投稿できる体制）。

08 / KENSEKI

見積自動化（積算）深掘り — 商業施設・公共工事の見積を Claude Code で半自動化する

商業施設・公共工事の見積業務は、複数の協力会社・専門工事業者から届く見積（PDF・Excel・FAX・手書き・写真）を束ねて自社の積算書に統合する仕事。1円の計算ミスが利益を吹き飛ばし、入札では失格になる領域です。本章では、Claude Codeエージェントで見積作成を半自動化する5つの設計パターン（K1〜K5）と、避けて通れない15の計算ミス防止チェック、公共工事・商業施設特有の論点までを解説します。

本章の最重要原則：AIに四則演算をさせない。
AIに「12,500 × 35.7 = ?」を解かせると、確率的に間違えます。AIは「読み取り・分類・転記・整合検査」担当、計算はExcel数式・Python・専用システム担当。この役割分担を守ったときだけ、AI見積自動化は安全に動きます。
SaaS（ミャク楽・AI積算）と Claude Code 自前構築の棲み分け：競合ではなく組み合わせる。
SaaS の強み：建材データベース・標準歩掛り・OCR精度が箱で揃う。サブスクで即効・全社導入も早い。
Claude Code 自前の強み：① 自社固有の積算ルール・属人ノウハウを直接スキル化 ② 機密案件・公共工事の入札資料をローカル処理（社外送信なし） ③ SaaS非対応の工種・特殊建材を補完。
→ 標準業務の80%をSaaSで時短し、残り20%の特殊・機密領域を Claude Codeでカバーするハイブリッド構成が現実解。

K1〜K5：見積自動化の5パターン

K1 / COLLECT

業者見積の取り込み・統合スキル /mitsumori-collect

核：協力会社・専門工事業者から届く形式バラバラの見積（PDF／Excel／写真／手書き）を、Claude Codeが社内標準フォーマットに変換。AIの仕事は抽出・分類・転記まで。計算は触らない。
このスキルで実現できること
  • 10社から届いた見積PDFをフォルダに入れる → 1コマンドで統合Excelに転記
  • 工種コード（建築・電気・設備・外構）の自動分類
  • 業者ごとに違う品名表記を「自社標準辞典」で正規化（例：「コンクリート」「RC」「無筋C」→ 統一）
  • 抽出した数値1つ1つに 「原本PDFのどこに書かれていたか」のページ番号・位置参照を付与
  • 不明・読取不能の項目は「要人手確認」フラグを立てて止める（推測で埋めない）
実装プロセス
  1. 業者見積のサンプル10枚をClaude Codeに渡し、共通項目を抽出させる
  2. 「品名・規格・数量・単位・単価・金額・備考」の7列テンプレを確定
  3. /mitsumori-collect スキルを生成（A1参照）。Excelテンプレに数式を仕込み、AIは値だけ書き込む
  4. 10案件分テスト → 精度95%超を確認してから運用へ
参考実装サービス
  • ミャク楽（2026/3公開）：手書き・写真・PDF対応の建設特化AI-OCR。階層構造を維持してNET金額・掛率を自動計算
  • AI-OCR 佐助：複雑レイアウト・手書きPDF → Excel変換
  • 自社開発するなら：Claude Code＋OCRライブラリ（Tesseract/Cloud Vision API等）
K2 / CHECK

3視点並列バリデーション @mitsumori-checker

核：K1で集約した見積データを 3つの独立エージェントで同時検査。1人で順番に見ていたら見落とす不整合を、並列で炙り出す（A2応用）。
3エージェントの役割分担
  • AGENT A：単位整合チェッカー — ㎡／㎥／m／本／組／式／箇所の混在を検出。「同じ品目で業者A=㎡・業者B=㎥」を警告
  • AGENT B：単価相場チェッカー — 過去案件と国交省「公共建築工事標準単価積算基準」を照らし、相場から ±X% 超は赤フラグ
  • AGENT C：計算検算チェッカー — 横計（数量×単価=金額）・縦計（小計の合計）・総額の三方向を Excel関数で再計算し、表記値とズレたら警告
運用イメージ
  1. K1の出力Excelをサブエージェント3体に同時送付
  2. 各エージェントが独立コンテキストで検査
  3. 結果を統合 → 「要対応」リストのみが返ってくる。OK項目は流す
  4. 代表は不整合だけ確認すればよく、全件目視より5〜10倍速い
K3 / HIROI

数量拾い直し＆突き合わせ /suuryouhiroi-check

核：設計図書PDFから自社で数量を独自に拾い直し、業者見積と突き合わせる。業者の「拾い忘れ」「過剰計上」を機械的に検出する保険。
検出できる典型問題
  • 拾い忘れ：業者見積に含まれていない部位（梁・小庇・ピット・捨てコン等）の発見
  • 過剰計上：窓・建具・開口部の差引漏れ（壁面積から開口分を引いていない）
  • 仕様違い：設計図はFc=24なのに見積はFc=21、など強度・等級・厚みのズレ
  • 範囲違い：テナント工事と躯体工事の境界をまたいだ二重計上
実装パターン（実務優先順）
  1. CAD/BIMからの直接拾い（最優先）：Jw_cad・Vectorworks・Revit・ArchiCAD・SXF（電子納品）からの数量取得はPDF OCRより精度・速度ともに圧倒的に上。CAD前提の案件はまずこれを検討
  2. AI積算SaaS連携：「AI積算（aisekisan.com）」「KK Generation」「ミャク楽」等の専門サービスから拾い数量を取得
  3. 自前実装：CAD/BIM未対応の旧紙図面案件に限り、Claude CodeがPDF図面を読み、部屋・面積・建具を識別 → Excelに数量出力
  4. K1の業者見積データと Excel上で機械的に比較（VLOOKUP・SUMIFS等）、差分をハイライト
K4 / MEMORY

過去案件Memory見積参謀 /mitsumori-rag

核：過去案件の成約価格・実行原価・利益率・業者別単価をMemoryに蓄積。新規案件で「類似案件はこの単価で通った」「B社のほうが3%安い」を即提示。AIに創作させない＝過去データから引用させるのがハルシネーション対策の本丸。
蓄積するデータ
  • memory/案件別実績DB.md — 案件名・延床・工種・成約見積・実行原価・利益率
  • memory/業者別単価履歴.md — 業者ごとの工種別単価推移（年単位）
  • memory/失敗事例.md — 「あの案件で読み落としたミス」「業者から後出しで上がった項目」
  • memory/補助金・公共工事ルール.md — 入札・補助金で通る／通らないの境界線
NotebookLMで簡易構築する場合
  1. 過去5年の見積書・契約書・実行原価表PDFを50〜100本投入
  2. 「100㎡規模・商業施設・内装込みの過去案件は？」と日本語で問う
  3. 該当案件の単価・諸経費率が即時表示される（Claudeに引用範囲を明示させる）

参考：ChatGPTで見積書を作る事例で「30分→5分短縮」報告あり。NotebookLMで過去案件50件投入の運用例（株式会社Lib）

K5 / FINAL

最終見積書ジェネレータ＋検算フック /mitsumori-final

核：K1〜K4の結果を 顧客提出用フォーマットに組み上げる最終段階。各段階で Hooks が自動発火して検算し、出力前に 全項目の二重検算 を強制。
処理フロー
  • STEP 1：内訳明細書（K1+K3を統合）
  • STEP 2：工種別集計（建築・電気・設備・外構・諸経費）
  • STEP 3：諸経費載せ（共通仮設費・現場管理費・一般管理費の3層）
  • STEP 4：自社利益率上乗せ（案件別・施主別ルール）
  • STEP 5：顧客提出フォーマット整形（公共工事様式／民間様式／英文）
Hooksで仕込む自動検算
  1. ファイル保存時 → 縦計・横計・小計の三方向検算スクリプトを自動実行
  2. 不整合があれば即「⚠ 検算エラー」を画面に表示してロック
  3. 承認ボタンは代表のみが押せる（権限制御）
  4. 承認後に 提出版_YYYYMMDD_HHmm.xlsx として版管理保存

★ Excelテンプレに SUMPRODUCT・ROUNDDOWN・IF 等の数式を最初から仕込んでおき、AIは「値の埋め込み」しかしない設計が安全

計算ミスを起こさない 15 のチェックポイント

★マーク（赤枠）は絶対に外してはいけない最重要項目。残り12項目は運用ルール化推奨。

CRITICAL 01
★ AIに四則演算をさせない

AIは「12,500 × 35.7」を確率的に間違える。計算は必ずExcel関数・Python・専用システムに渡す。AIの担当は読み取り・分類・転記・整合検査まで。

CRITICAL 02
★ 単位を必ず明示・統一

㎡／㎥／m／本／組／式／箇所／kg／t を必ず明記。「自社標準単位辞典」を作って、業者間の表記揺れを正規化。

CRITICAL 03
★ 原本との位置参照を残す

抽出した数値1つ1つに、PDFのページ番号・位置（XY座標）を付与。「あの数字どこから来た？」が3秒で辿れる状態を作る。

CRITICAL 04
★ 三方向検算＋歩掛り検算

「数量×単価=金額」「小計の合計=工種計」「工種計の合計=総額」の3方向を Excel関数で再計算してロック。加えて「労務費＝人工数×職種別単価×歩掛り」の妥当性も別軸でチェック（労務費の歩掛りズレは現場で初めて顕在化する典型ミス）。

CRITICAL 05
★ 人間（代表）の最終承認を必修化

どんなに精度が高くても、AIが最終ボタンを押せない設計に。承認権限は代表のみ（CC の allow 機能で権限制御）。

CHECK 06
桁チェック（100万 vs 1,000万）

OCR誤読で1桁ズレる致命的事故を防ぐため、「過去同等案件の±50%超は警告」「金額が3桁以上跳ねたら停止」を自動化。

CHECK 07
端数処理ルールを明文化

円未満切り捨て／諸経費は1,000円単位／消費税10%（軽減税率は別計算）／合計の千円切捨。関数（ROUNDDOWN等）を必ず固定。

CHECK 08
同名異義の検出

「コンクリート」「RC」「無筋C」、「鉄筋」「SD295」「異形棒鋼」など、業界用語の表記揺れを辞典で名寄せ。違う品目が同じ名前で書かれる事故を防ぐ。

CHECK 09
過去案件からの逸脱検知

K4のMemoryと照合し、「この工種、過去平均より40%高い／低い」を機械的に警告。異常値の見落としをゼロに。

CHECK 10
複数業者比較で最安値即採用しない

同じ工種に複数業者の見積が来た場合、最安値を機械的に採用すると品質・工期・後出し追加のリスク。「最安値だが要警戒」のフラグを残す。

CHECK 11
業者からの差替え版を版管理

業者から「修正版です」と再見積が来たら、必ず旧版とdiffを取る。Claude Code＋gitで自動差分管理（AS資産の /doc-compare 流用可能）。

CHECK 12
OCR誤読の典型を辞書化

「0」と「O」、「1」と「I」、「,」と「.」、「3」と「8」、「5」と「6」など、OCR誤読しやすい組合せを事前辞書化。読取後に再走査。

CHECK 13
Excel数式を壊さない貼り付け

AIが値を入れる時、「値の貼り付け」モードに限定し、数式セルには直接書き込ませない。テンプレ駆動で「数式エリア／値エリア」を物理分離。

CHECK 14
Hooks の無限ループ防止

「保存→検算→修正→保存→…」が無限に回らないよう、Hook内で「最終更新時刻と一致したらスキップ」「再帰深さ上限」を必ず設定。

CHECK 15
テンプレ駆動・自由記述させない

毎回違うプロンプトで毎回違う結果が出ると検算できない。SKILL.mdに固定プロンプトを書き、入力欄だけが変わる設計に。

公共工事・商業施設の特有論点

標準歩掛り・積算基準への適合 国土交通省 大臣官房官庁営繕部の 「公共建築工事標準単価積算基準」（最終改定：令和8年3月11日 国営積第15号）および 「公共建築工事積算基準」関連資料 に準拠した諸経費率・歩掛り計算。都道府県ごとの上乗せ規定もK4のMemoryで管理。発注者（国・自治体・独法）別の差を機械的に判別。
共通仮設費・現場管理費・一般管理費 3層構造の諸経費率は工事規模・工種で変動。「直接工事費 → 共通仮設費 → 現場管理費 → 一般管理費」の積み上げを 必ず官公庁基準の率テーブル と突き合わせ。
VE（Value Engineering）提案 入札時の代替案・コストダウン提案。K4のMemoryから「過去にVE採用された代替工法・採用率」を引き出して提案ドラフト化。
テナント工事・躯体工事の境界 商業施設で頻発する「ここは躯体？ここはテナント？」の二重計上・抜け漏れ。K3の数量拾い直しで境界を機械的に判別する設計が有効。
消費税（軽減税率／免税） 原則10%だが、補助金事業・海外取引・一部材料で異なる。税率は必ず別列で明示し、合計式は SUMPRODUCT(数量, 単価)*(1+税率) で固定。
入札・契約書連動 公共工事は入札書類との完全一致が必須。AS資産の /contract-check と組み合わせ、見積書と契約書の品名・数量・単価の機械突き合わせを実施。
追加変更工事（赤伝・設計変更） 公共工事で必ず発生する「設計変更による追加・減額」（通称：赤伝）。当初契約 vs 変更後の差分を Claude Code で機械抽出し、変更協議書のドラフトまで自動化。AS資産の /doc-compare 流用可能。
下請法・建設業法 下請業者への支払サイト・書面交付・割引手形等の規制を遵守。K4の業者別履歴で 支払期日違反・歩掛り低すぎ等の警告を機械化。下請契約書のレビューもAS資産 /contract-check で対応。
インボイス制度（適格請求書） 2023年10月開始。免税業者からの仕入は仕入税額控除が段階的に削減（〜2029年）。業者ごとの登録番号有無を K4のMemoryで管理し、見積取得時に税額計算ロジックを切り分け。

PHASE 3 / 2ヶ月

PHASE 4 / 3〜6ヶ月
